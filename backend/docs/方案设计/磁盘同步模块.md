# 磁盘同步模块方案设计

**日期**：2026-06-22

## 一、需求概述

实现磁盘同步管理功能，支持：
- 查看主盘/备盘状态
- 获取/更新同步任务配置
- 手动触发同步
- 查看同步执行日志
- 后台定时自动同步（增量/全量）

同时实现 SyncWorker 后台同步工作器：
- 按配置的周期检查备盘状态并执行同步
- 增量模式：仅复制新文件/变更文件
- 全量模式：全量复制
- 同步失败时创建告警记录

## 三、数据模型

### 复用已有模型

**StorageDisk**（storage_disk 表）：
| 字段 | 类型 | 说明 |
|------|------|------|
| DiskType | int8 | 1=主盘, 2=备盘 |
| Status | int8 | 1=正常, 0=异常 |

**StorageSyncTask**（storage_sync_task 表）：
| 字段 | 类型 | 说明 |
|------|------|------|
| SyncMode | int8 | 1=增量, 2=全量 |
| CronExpr | string | cron 表达式 |
| IgnoreSuffix | string | 逗号分隔的忽略后缀 |
| SpeedLimit | *int64 | 限速（字节/秒），0=不限 |
| LastSyncTime | *time.Time | 上次同步时间 |
| LastSyncResult | *int8 | 1=成功, 2=失败 |
| IsRunning | int8 | 0=空闲, 1=运行中 |

**SysWarnLog**（sys_warn_log 表）：
| 字段 | 类型 | 说明 |
|------|------|------|
| WarnType | int8 | 3=磁盘同步告警 |

## 四、Handler 设计：StorageHandler

依赖：`StorageService` + `*gorm.DB`

```go
type StorageHandler struct {
    StorageService *service.StorageService
    DB             *gorm.DB
}
```

### 路由设计（全部在 storageGroup 下，已有 AdminRequired）

| 方法 | 路径 | Handler | 说明 |
|------|------|---------|------|
| GET | /api/storage/disk_info | DiskInfoHandler | 主备盘状态 |
| GET | /api/storage/disk_list | DiskListHandler | 存储盘列表（保留现有逻辑） |
| GET | /api/storage/quota | QuotaHandler | 用户配额（保留现有逻辑） |
| GET | /api/storage/sync_task | SyncTaskHandler | 获取同步配置 |
| PUT | /api/storage/sync_task | UpdateSyncTaskHandler | 更新同步配置 |
| POST | /api/storage/sync/manual | ManualSyncHandler | 手动触发同步 |
| GET | /api/storage/sync/logs | SyncLogsHandler | 同步日志（分页） |

### 响应结构体

```go
type DiskInfoResp struct {
    ID            int64  `json:"id"`
    DiskType      int8   `json:"diskType"`
    DiskPath      string `json:"diskPath"`
    TotalSize     int64  `json:"totalSize"`
    UsedSize      int64  `json:"usedSize"`
    AvailableSize int64  `json:"availableSize"`
    Status        int8   `json:"status"`
}

type SyncTaskResp struct {
    ID             int64  `json:"id"`
    SyncMode       int8   `json:"syncMode"`
    CronExpr       string `json:"cronExpr"`
    IgnoreSuffix   string `json:"ignoreSuffix"`
    SpeedLimit     int64  `json:"speedLimit"`
    LastSyncTime   string `json:"lastSyncTime"`
    LastSyncResult int8   `json:"lastSyncResult"`
    IsRunning      int8   `json:"isRunning"`
}

type UpdateSyncTaskReq struct {
    SyncMode     *int8   `json:"syncMode"`
    CronExpr     *string `json:"cronExpr"`
    IgnoreSuffix *string `json:"ignoreSuffix"`
    SpeedLimit   *int64  `json:"speedLimit"`
}
```

### ManualSyncHandler 流程
1. 检查是否已有同步任务在运行（is_running=1），是则返回"同步进行中"
2. 设置 is_running=1
3. 启动 goroutine 执行同步（与 SyncWorker 共用同步逻辑）
4. 返回"同步已触发"

### SyncLogsHandler 流程
1. 从 storage_sync_task 表分页查询（按 update_time 倒序）
2. 支持 page/pageSize 参数

## 五、Worker 设计：SyncWorker

依赖：`DB` + `StorageService` + `config.Config`

```go
type SyncWorker struct {
    DB             *gorm.DB
    StorageService *service.StorageService
    Config         *config.Config
}

func (w *SyncWorker) Start()
func (w *SyncWorker) executeSync() error
func (w *SyncWorker) doIncrementalSync(mainDisk, backupDisk *model.StorageDisk, ignoreSuffixes []string) error
func (w *SyncWorker) doFullSync(mainDisk, backupDisk *model.StorageDisk, ignoreSuffixes []string) error
func (w *SyncWorker) createWarnLog(content string)
```

### Start 流程
1. 获取同步任务配置
2. 使用 `time.Ticker` 按 cron 表达式周期执行
3. 每次 tick 时调用 `executeSync()`

### executeSync 流程
1. 查询备盘（disk_type=2），检查 status=1，不可用则记录告警并返回
2. 从 storage_sync_task 获取最新配置
3. 设置 is_running=1
4. 根据 sync_mode 执行增量/全量同步
5. 更新 last_sync_time、last_sync_result
6. 设置 is_running=0
7. 失败时创建 SysWarnLog（warn_type=3）

### 同步逻辑
- 增量同步：对比主盘和备盘文件，复制主盘有但备盘没有或更新的文件
- 全量同步：递归遍历主盘所有用户目录，复制全部文件到备盘
- 忽略 ignore_suffix 指定的后缀文件
- 使用 `io.Copy` 复制文件（与现有 copyFile 一致）

## 六、Router 变更

### 变更前
```go
storageGroup := r.Group("/api/storage")
storageGroup.Use(middleware.AdminRequired())
{
    storageGroup.GET("/disk_list", func(c *gin.Context) { ... }) // 内联 handler
}
storageGroup.GET("/quota", func(c *gin.Context) { ... })         // 内联 handler
```

### 变更后
创建 StorageHandler 实例，替换所有内联 handler：

```go
storageHandler := &handler.StorageHandler{
    StorageService: storageService,
    DB:             db,
}

storageGroup := r.Group("/api/storage")
storageGroup.Use(middleware.AdminRequired())
{
    storageGroup.GET("/disk_list", storageHandler.DiskListHandler)
    storageGroup.GET("/disk_info", storageHandler.DiskInfoHandler)
    storageGroup.GET("/quota", storageHandler.QuotaHandler)
    storageGroup.GET("/sync_task", storageHandler.SyncTaskHandler)
    storageGroup.PUT("/sync_task", storageHandler.UpdateSyncTaskHandler)
    storageGroup.POST("/sync/manual", storageHandler.ManualSyncHandler)
    storageGroup.GET("/sync/logs", storageHandler.SyncLogsHandler)
}
```

## 七、main.go 变更

```go
// 启动后台清理工作器
(&worker.CleanupWorker{DB: db}).Start()

// 启动磁盘同步工作器
(&worker.SyncWorker{
    DB:             db,
    StorageService: storageService,
    Config:         cfg,
}).Start()
```

注意：需要在 main.go 中增加 storageService 初始化（目前已在 router.go 中初始化，需提取到 main.go）。

## 八、技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 分层 | handler 直接持有 DB | 简洁实现，避免为简单 CRUD 新增 repo/service 层 |
| 定时任务 | time.Ticker + goroutine | 与 CleanupWorker 一致，无需引入 robfig/cron |
| 手动同步 | handler 内 goroutine + 共用 executeSync | 简单直接，共用同步逻辑 |
| 同步逻辑 | 在 SyncWorker 中实现 | 内聚在 worker 中，handler 手动触发时复用 |
| 依赖注入 | 构造函数外注入 | 与项目 router.go 中 DI 模式一致 |
| 分页 | offset/limit | 与现有 file handler 一致 |
| 错误处理 | 显式检查 error | Go 惯用法 |
